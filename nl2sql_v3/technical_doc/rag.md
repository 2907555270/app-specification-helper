# 检索增强生成（RAG）技术指南

本文档详细介绍 NL2SQL 系统中使用的检索技术，包括三种核心检索方式、重排模型原理及其在实际应用中的最佳实践。

---

## 一、三种检索方式概述

在 NL2SQL 及更广泛的 RAG（Retrieval-Augmented Generation）系统中，从大量表或文档中召回相关内容通常采用以下三种检索方式：

| 维度 | 关键词检索（BM25） | 稀疏向量检索（Sparse） | 稠密向量检索（Dense） |
|------|-------------------|----------------------|---------------------|
| **核心原理** | 词频统计 + 倒排索引 | 神经网络学习的词权重 | 语义向量表示 |
| **向量特点** | 传统无向量或极稀疏 | 高维极稀疏（数万维） | 低维稠密（768-4096维） |
| **匹配方式** | 精确/近似词匹配 | 加权词汇匹配 | 向量空间距离 |
| **语义理解** | 几乎无（字面匹配） | 弱至中 | 强（同义/隐含理解） |
| **OOV 处理** | 优（见词即匹配） | 中（依赖词表） | 较差（可能映射错误） |
| **计算成本** | 最低 | 中等 | 最高 |
| **典型代表** | BM25、Elasticsearch | SPLADE、BGE-M3 | Sentence-BERT、BGE |

### 1.1 各检索方式详解

#### 关键词检索（BM25）
- **原理**：基于词频统计和逆文档频率（IDF），通过倒排索引实现快速匹配
- **优势**：对表名、列名、ID、年份等确定性元素匹配精确
- **劣势**：无法处理同义词和改写表达
- **NL2SQL 适用场景**：精确匹配表名 `sales_2024`、列名 `user_id` 等

#### 稀疏向量检索（Sparse Embedding）
- **原理**：使用神经网络学习词汇重要性权重，输出高维稀疏向量
- **优势**：比传统 BM25 更智能，支持一定程度的相关词扩展
- **劣势**：对未见过的专业术语处理有限
- **NL2SQL 适用场景**：专业术语匹配、领域词汇识别

#### 稠密向量检索（Dense Embedding）
- **原理**：使用深度学习模型将文本编码为稠密向量，通过向量相似度匹配
- **优势**：强大的语义理解能力，能处理自然语言查询和同义改写
- **劣势**：对专有名词、数字、ID 等精确匹配较弱
- **NL2SQL 适用场景**：自然语言描述理解、复杂语义查询

### 1.2 组合策略建议

根据 2025-2026 年的行业共识，推荐采用**多路召回 + 融合重排**策略：

```
┌─────────────────────────────────────────────────────────────┐
│  阶段一：多路粗召回（并行执行）                              │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐           │
│  │ BM25 召回   │ │ Sparse 召回 │ │ Dense 召回  │           │
│  │ (保底精确)  │ │ (词汇增强)  │ │ (语义理解)  │           │
│  └──────┬──────┘ └──────┬──────┘ └──────┬──────┘           │
│         └───────────────┼───────────────┘                   │
│                         ▼                                   │
│  阶段二：分数融合（RRF 或加权）                             │
│                         ▼                                   │
│  阶段三：重排（Cross-Encoder 可选）                        │
│                         ▼                                   │
│  输出：Top-N 相关表                                         │
└─────────────────────────────────────────────────────────────┘
```

**配置建议**：
- 表 schema 规范、查询多精确匹配 → BM25 + Sparse
- 表命名随意、查询偏自然语言 → BGE-M3（Sparse + Dense 双输出）
- 追求最佳效果 → BGE-M3 Dense + Sparse 融合 + Cross-Encoder 重排

## 二、重排模型详解

### 2.1 为什么需要重排

粗召回阶段（BM25 + 向量检索）完成后，通常存在以下问题：
- 相关性高的表被埋没在排名中后部
- "看起来像但实际无关"的表排名靠前
- 语义向量压缩导致细节丢失

重排模型（Cross-Encoder）通过深度交叉注意力机制，解决**候选集内部的精细排序与噪声过滤**问题。

### 2.2 Bi-Encoder vs Cross-Encoder

| 维度 | Bi-Encoder（双编码器） | Cross-Encoder（交叉编码器） |
|------|----------------------|---------------------------|
| **输入方式** | Query 和 Document 分别独立编码 | Query + Document 拼接后联合编码 |
| **交互时机** | Early Interaction（几乎无交互） | Full Interaction（深度交叉） |
| **信息捕捉** | 独立语义向量，依赖向量比较 | 每个 Token 互相可见 |
| **速度** | 快（向量可预计算 + ANN 索引） | 慢（需逐对计算） |
| **精度** | 中等 | 显著更高（10-30% NDCG 提升） |
| **适用阶段** | 第一阶段粗召回（百万→百） | 第二阶段精排（百→十） |

**通俗理解**：
- Bi-Encoder：各自画个向量，然后比距离
- Cross-Encoder：把 Query 和 Document 放在一起"深度交流"，然后打分

### 2.3 重排能捕捉的语义信息

Cross-Encoder 能精细处理以下情况：
- **同义改写**：`"clubs"` ≈ `"社团"` ≈ `"ClubName"`
- **否定表达**：Query 有 `"not in sports"` 时压低体育相关表
- **细粒度匹配**：问 `"club leader"` 时偏向有 `Member_ID/Year_Join` 的表
- **整体意图一致性**：综合考虑而非只看关键词

## 三、ColBERT 详解

### 3.1 什么是 ColBERT

**ColBERT**（Contextualized Late Interaction over BERT）由斯坦福大学于 2020 年提出，是介于 Bi-encoder 和 Cross-encoder 之间的检索模型。

它通过 **Late Interaction（晚期交互）** 机制实现了精度与效率的平衡：

```
┌─────────────────────────────────────────────────────────────┐
│  编码阶段（可离线预计算）                                   │
│  Query: "销售数据" ──→ [vec₁, vec₂, vec₃, ...]           │
│  Doc: "2024年销售额报表" ──→ [vec₁, vec₂, ..., vecₙ]    │
├─────────────────────────────────────────────────────────────┤
│  查询阶段（在线计算）                                       │
│  对每个 Query token，找 Doc 中最相似的 Token               │
│  MaxSim(q, d) = Σ max_sim(qi, dj)                         │
├─────────────────────────────────────────────────────────────┤
│  优势                                                       │
│  ✓ 比 Bi-encoder 精度高（接近 Cross-encoder）             │
│  ✓ 比 Cross-encoder 速度快（向量预计算）                   │
│  ✓ 保留 Token 级别细粒度匹配                               │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 ColBERT vs 其他方案对比

| 方案 | 编码方式 | 交互时机 | 精度 | 效率 | 典型模型 |
|------|---------|----------|------|------|---------|
| BM25 | 无神经网络 | 无 | 中 | 极快 | Elasticsearch |
| Bi-Encoder | 单向量 | Early | 中高 | 快 | BGE-M3 Dense |
| Cross-Encoder | 拼接 | Full | 极高 | 慢 | BGE-Reranker |
| ColBERT | 多向量 | Late | 高极高 | 中快 | Jina-ColBERT-v2 |

### 3.3 在 NL2SQL 中的应用建议

```
推荐流程：
1. BM25 / BGE-M3 Sparse → 召回 Top-100~200
2. ColBERT / Cross-Encoder → 重排 Top-20~50
3. 最终输出 Top-5~10 给 LLM
```

**硬件要求**：RTX 4060 可本地运行 ColBERT（压缩版约 1-2GB VRAM）

# RRF
**RRF（Reciprocal Rank Fusion，互逆排名融合）** 是目前信息检索和混合搜索（hybrid search）中最常用、最鲁棒的融合方法之一，尤其在结合多种召回方式（BM25、sparse、dense、knn 等）时，效果往往比手动加权分数更好。

下面用通俗的方式完整介绍一下它的原理、为什么好用、计算公式，以及在 NL2SQL 场景中的实际意义。

### 1. RRF 的核心思想（一句话总结）

**不看原始分数，只看每个召回源给文档的排名位置，然后用一个简单的公式把多个排名的“倒数”加起来，谁的总分高谁就排前面。**

它假设：
- 每个召回源（retriever）都有自己的“靠谱程度”
- 但原始分数尺度差异很大（BM25 可能几百，dense cosine 0~1，sparse 权重又不同）
- 排名顺序比绝对分数更可靠

所以干脆抛弃分数，只利用“谁排第几”的信息。

### 2. RRF 的计算公式

对一个文档 d，在多个检索器（retrievers）上的排名融合分数是：

```
RRF(d) = Σ (over all retrievers r)   1 / (k + rank_r(d))
```

- rank_r(d)：文档 d 在检索器 r 中的排名位置（第 1 名 = 1，第 2 名 = 2，...，没召回 = 不加）
- k：一个常数（hyperparameter），通常取 60（论文和实践中最常见的默认值）
- Σ：所有检索器的贡献相加

最终按 RRF(d) 从大到小排序，得到融合后的排名。

### 3. 举个最直观的例子

假设有 3 个检索器（BM25、Sparse、Dense），对同一个 query 召回了以下文档：

| 文档       | BM25 排名 | Sparse 排名 | Dense 排名 |
|------------|-----------|-------------|------------|
| club_1     | 1         | 3           | 5          |
| club_leader| 3         | 1           | 2          |
| yelp       | 2         | 8           | 1          |
| baseball_1 | 10        | 15          | 20         |

用 k = 60 计算 RRF 分数：

- club_1：  
  1/(60+1) + 1/(60+3) + 1/(60+5) = 1/61 + 1/63 + 1/65 ≈ 0.01639 + 0.01587 + 0.01538 ≈ **0.04764**

- club_leader：  
  1/(60+3) + 1/(60+1) + 1/(60+2) = 1/63 + 1/61 + 1/62 ≈ 0.01587 + 0.01639 + 0.01613 ≈ **0.04839**

- yelp：  
  1/(60+2) + 1/(60+8) + 1/(60+1) = 1/62 + 1/68 + 1/61 ≈ 0.01613 + 0.01471 + 0.01639 ≈ **0.04723**

- baseball_1：  
  1/(60+10) + 1/(60+15) + 1/(60+20) ≈ 很小 ≈ **0.013**

**最终排序**：club_leader > club_1 > yelp > baseball_1

可以看到：
- club_leader 在 Sparse 排第 1、在 Dense 排第 2，虽然 BM25 只排第 3，但综合分数最高
- yelp 在 Dense 排第 1，但其他源排名很差，被大幅拉低
- 这正是我们想要的：多路互补，综合最可靠的排名

### 4. RRF 为什么在实践中这么好用？（主要优势）

| 优势点                     | 解释                                                                 | 对 NL2SQL 的帮助 |
|----------------------------|----------------------------------------------------------------------|------------------|
| 不依赖分数尺度             | BM25、dense、sparse 分数单位完全不同，RRF 忽略绝对值，只看相对顺序 | 不用再手动调 weights 了 |
| 鲁棒性极强                 | 一个检索器出错（召回假高分），影响很小，因为排名靠后贡献就小       | 噪声表不会轻易冲到前面 |
| 实现简单、参数少           | 只有一个 k（默认 60 基本够用），调参成本低                         | 比手动 weights 省事 |
| 常优于加权线性融合         | 学术和工业 benchmark 中，RRF 经常比手动加权高 5–15% NDCG/MRR       | Hit@1 和 MRR 容易再涨几个点 |
| 支持加权（weighted RRF）   | 可给不同检索器不同权重（如 sparse 给 1.5，dense 给 1.0）           | 保留你偏好 sparse 的倾向 |

### 5. 在你的 NL2SQL 场景中的实际意义

从你之前的测试数据看：
- 当前手动 weights（keyword 0.1 / sparse 0.6 / dense 0.3）已经不错，但调参过程痛苦
- 引入 RRF 后，可以：
  - 直接把三路召回结果（BM25、sparse、dense）分别跑 Top-200
  - 用 RRF 融合 → 得到更稳定的 Top-100
  - 再 rerank 到 Top-5~10

预期效果：
- Hit@1 可能从 0.81–0.88 再向上浮动 3–10%（尤其在 Not Determined Scope 时更明显）
- 减少对 weights 的敏感性，系统更稳定
- 延迟增加很少（融合计算是 O(n log n)，n=200 几乎忽略不计）

### 总结一句话

**RRF 的原理就是“谁在多个检索器里都排得比较靠前，谁就赢”，通过把排名取倒数相加来融合，避免了分数尺度不一致和手动调权重的麻烦，是目前混合召回最实用、最推荐的融合方式。**
