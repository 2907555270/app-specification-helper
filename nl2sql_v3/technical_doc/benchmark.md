# 基准测试与评估

本文档详细介绍 NL2SQL 表召回系统的评估指标、测试方法及性能分析结果。

---

## 一、评估指标说明

### 1.1 Hit Rate @ K

**Hit Rate @ K**（命中率）表示在召回的 Top-K 结果中，**至少包含一个正确相关项**的查询占比。

**计算公式**：
```
Hit Rate @ K = (Top-K 内有正确项的查询数) / 总查询数
```

| 指标 | 含义 | 业务价值 |
|------|------|---------|
| **Hit@1** | Top-1 中有正确表 | 最高优先级，第一个结果必须准确 |
| **Hit@3** | Top-3 中有正确表 | 允许 LLM 从 3 个候选中选择 |
| **Hit@5** | Top-5 中有正确表 | 兜底保障，覆盖绝大多数场景 |

### 1.2 Mean Reciprocal Rank (MRR)

**MRR**（平均倒数排名）衡量**第一个正确项的平均排名位置**，强调正确项应尽量靠前。

**计算公式**：
```
对于单个查询：RR = 1 / rank（第一个正确项的位置，无召回则 RR = 0）
MRR = Σ RR / 总查询数
```

**特点**：
- 范围：0 ~ 1
- 排名越靠前得分越高：第1名 = 1分，第2名 = 0.5分，第5名 = 0.2分
- 对排名靠后情况惩罚更严重

## 三、深度分析

### 3.1 数据库范围的影响

| 场景 | Hit@1 | MRR | 差距 |
|------|-------|-----|------|
| Determined | 0.76–0.88 | 0.83–0.93 | — |
| Not Determined | 0.36–0.67 | 0.47–0.77 | 显著下降 |

**结论**：预先缩小查询范围（如通过用户选择、历史偏好）是提升召回质量的**最有效手段**。全库搜索时噪声表过多，语义向量容易被稀释。

### 3.2 权重配置的影响（无重排对比）

| 权重配置 | Determined Hit@1 | Not Determined Hit@1 | 速度变化 |
|----------|------------------|---------------------|----------|
| sparse:0.5, dense:0.4 | 0.76 | 0.36 | 基准 |
| sparse:0.6, dense:0.3 | 0.81 | 0.67 | +26% |

**发现**：提升 Sparse 权重、降低 Dense 权重效果显著：
- Determined 场景：Hit@1 提升 5.4%
- Not Determined 场景：Hit@1 提升 **84%**

**原因分析**：
- 表名/列名中缩写、专有名词多
- Sparse 向量对专业术语更鲁棒

### 3.3 重排的效果与代价

**精度提升（Determined 场景）**：
| 指标 | 无重排 | 有重排 | 提升 |
|------|--------|--------|------|
| Hit@1 | 0.8137 | 0.8756 | +7.6% |
| Hit@3 | 0.8798 | 0.9898 | +12.5% |
| Hit@5 | 0.8803 | 0.9958 | +13.1% |
| MRR | 0.8448 | 0.9317 | +10.3% |

**延迟代价**：
| 场景 | 平均耗时 | 增幅 |
|------|----------|------|
| Determined | 0.046s → 0.085s | +85% |
| Not Determined | 0.042s → 0.197s | +370% |

**结论**：
- 重排适用于已确定范围的场景
- 全库搜索时延迟代价过大，建议限制重排规模（如 Top-80~100）

## 四、优化建议

### 4.1 推荐配置

```
权重配置：
├── keyword: 0.1
├── sparse:  0.6  ← 主要依赖
└── dense:   0.3

重排策略：
├── Determined 场景：启用重排
└── Not Determined 场景：重排 Top-100
```

**预期性能**：
- Hit@1 ≈ 0.87–0.88
- Hit@5 ≈ 0.99+
- 平均耗时 ≈ 0.08s/查询

### 4.2 短期优化方向

| 优化点 | 说明 |
|--------|------|
| 范围预判 | 先用轻量分类器判断查询所属领域/数据库 |
| 重排规模 | 限制输入规模，只重排 Top-80~100 |
| 批量处理 | 调大 batch_size（RTX 4060 支持 32-64） |
| 权重微调 | 尝试 keyword:0.15 / sparse:0.65 / dense:0.2 |

### 4.3 长期优化方向

| 优化点 | 说明 |
|--------|------|
| RRF 融合 | 用 Reciprocal Rank Fusion 代替手动权重 |
| 端到端评估 | 引入 Exec Acc（SQL 执行正确率）指标 |
| 知识图谱 | 引入外键关系，增强多表查询能力 |
| Fine-tuning | 用 Spider/BIRD 数据集微调 LLM |

## 五、系统优化点一览

| 类别 | 优化项 | 潜在收益 | 实现难度 |
|------|--------|----------|----------|
| **模型** | LLM Fine-tuning | +10-20% Exec Acc | 中 |
| **模型** | 多模型 Ensemble | 平衡成本与精度 | 高 |
| **数据** | Schema 摘要 | 减少 token 浪费 | 低 |
| **数据** | 知识图谱增强 | 提升多表查询 | 中 |
| **架构** | 并行召回 | -30~50% 延迟 | 中 |
| **架构** | 结果缓存 | 减少重复计算 | 低 |
| **效率** | 量化部署 | 降低推理成本 | 低 |

建议按优先级逐项 A/B 测试，以量化指标指导优化方向。
