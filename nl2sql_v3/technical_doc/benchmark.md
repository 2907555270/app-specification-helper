# 基准测试与评估

本文档详细介绍 NL2SQL 表召回系统的评估指标、测试方法及性能分析结果。

***

## 一、评估指标说明

### 1.1 Hit Rate @ K

**Hit Rate @ K**（命中率）表示在召回的 Top-K 结果中，**至少包含一个正确相关项**的查询占比。

**计算公式**：

```
Hit Rate @ K = (Top-K 内有正确项的查询数) / 总查询数
```

| 指标         | 含义          | 业务价值              |
| ---------- | ----------- | ----------------- |
| **Hit\@1** | Top-1 中有正确表 | 最高优先级，第一个结果必须准确   |
| **Hit\@3** | Top-3 中有正确表 | 允许 LLM 从 3 个候选中选择 |
| **Hit\@5** | Top-5 中有正确表 | 兜底保障，覆盖绝大多数场景     |

### 1.2 Mean Reciprocal Rank (MRR)

**MRR**（平均倒数排名）衡量**第一个正确项的平均排名位置**，强调正确项应尽量靠前。

**计算公式**：

```
对于单个查询：RR = 1 / rank（第一个正确项的位置，无召回则 RR = 0）
MRR = Σ RR / 总查询数
```

**特点**：

- 范围：0 \~ 1
- 排名越靠前得分越高：第1名 = 1分，第2名 = 0.5分，第5名 = 0.2分
- 对排名靠后情况惩罚更严重

## 三、深度分析

### 3.1 数据库范围的影响（最关键因素）

| 配置类型 | DB Scope | Hit@1 | MRR | 与 Determined 差距 |
| --- | --- | --- | --- | --- |
| 无重排 (最佳权重 0.6/0.3) | Determined | 0.8137 | 0.8448 | —   |
| 无重排 (最佳权重 0.6/0.3) | Not Determined | 0.6707 | 0.7749 | ↓ 17.5% / ↓ 8.3% |
| 有重排 | Determined | 0.8756 | 0.9317 | —   |
| 有重排 | Not Determined | 0.6060 | 0.7318 | ↓ 30.8% / ↓ 21.5% |

**核心结论**：  
预先确定数据库范围是提升召回质量的**最大杠杆**。全库搜索时噪声表极多，导致语义向量被严重稀释，Hit@1 下降幅度可达 30%+。

### 3.2 权重配置的影响（无重排场景）

| 权重配置 | Determined Hit@1 | Not Determined Hit@1 | 主要收益场景 |
| --- | --- | --- | --- |
| keyword:0.1 / sparse:0.5 / dense:0.4 | 0.7597 | 0.3638 | —   |
| keyword:0.1 / sparse:0.6 / dense:0.3 | 0.8137 | 0.6707 | 全库搜索获益最大 |

**关键发现**：

- 提升 sparse 权重、降低 dense 权重在**全库场景**下效果极为显著（Hit@1 +84%）
- 原因：表名/字段名中存在大量缩写、专有名词、领域术语，**sparse 向量（SPLADE 等）对这些离散特征更鲁棒**，而 dense 向量在噪声环境下容易漂移。

### 3.3 重排 + Weighted RRF 的效果与代价

**精度对比（Determined 场景）**：

| 配置  | Hit@1 | Hit@3 | Hit@5 | MRR | 相对无重排提升 |
| --- | --- | --- | --- | --- | --- |
| 无重排 (0.6/0.3) | 0.8137 | 0.8798 | 0.8803 | 0.8448 | —   |
| 有重排 | 0.8756 | 0.9898 | 0.9958 | 0.9317 | +7.6% / +10.3% |
| 有重排 + Weighted RRF | **0.8775** | **0.9930** | **1.0000** | **0.9341** | +7.9% / +10.6% |

**延迟代价**：

| 配置  | Determined 平均耗时 | Not Determined 平均耗时 | 增幅  |
| --- | --- | --- | --- |
| 无重排 | 0.046s | 0.042s | —   |
| 有重排 | 0.085s (+85%) | 0.197s (+370%) | 高   |
| 有重排 + Weighted RRF | 0.095s (+107%) | 0.242s (+476%) | 更高  |

**结论**：

- 重排在 Determined 场景下收益显著，Hit@5 可接近 1.0
- Weighted RRF 进一步微抬精度（Hit@5 达到完美 1.0），但延迟略有增加
- 全库场景下重排代价过高（平均耗时翻 4–5 倍），需严格限制重排规模

### 3.4 BGE-M3 向量模型升级效果

**模型切换**：SPLADE + all-MiniLM-L6-v2 → **BGE-M3**

| 场景 | 指标 | 原模型 | BGE-M3 | 提升幅度 |
| --- | --- | --- | --- | --- |
| **Determined** | Hit@1 | 0.8775 | **0.8989** | +2.4% |
| **Determined** | Hit@3 | 0.9930 | 0.9925 | -0.05% |
| **Determined** | Hit@5 | 1.0000 | 0.9981 | -0.2% |
| **Determined** | MRR | 0.9341 | **0.9457** | +1.2% |
| **Not Determined** | Hit@1 | 0.5724 | **0.6334** | **+10.7%** |
| **Not Determined** | Hit@3 | 0.7834 | **0.8482** | +8.3% |
| **Not Determined** | Hit@5 | 0.8528 | **0.9092** | +6.6% |
| **Not Determined** | MRR | 0.6958 | **0.7539** | +8.4% |

**关键发现**：

- **BGE-M3 在全库搜索场景下提升显著**，Not Determined 的 Hit@1 从 57% 提升至 63%，这是因为 BGE-M3 的多语言能力 + dense-sparse 混合检索模式对复杂表名、领域术语更具鲁棒性
- Determined 场景也有小幅提升，Hit@1 从 87.75% 提升至 89.89%
- 延迟略有增加（约 5%），但在可接受范围内

## 四、优化建议

### 4.1 当前最佳推荐配置

```
权重配置（无重排 / 基础版）：
├── keyword: 0.1
├── sparse:  0.6   ← 主力召回
└── dense:   0.3

高级版（Determined 场景）：
├── 召回：keyword 0.1 / sparse 0.6 / dense 0.3
├── 向量模型：BGE-M3（推荐）
└── 融合：Weighted RRF (k=100) + 重排 Top-20

预期性能（Determined + BGE-M3）：
- Hit@1 ≈ 0.899
- Hit@5 ≈ 0.998
- MRR   ≈ 0.946
- 平均耗时 ≈ 0.100s

预期性能（Not Determined + BGE-M3）：
- Hit@1 ≈ 0.633
- Hit@5 ≈ 0.909
- MRR   ≈ 0.754
- 平均耗时 ≈ 0.324s
```

### 4.2 短期优化方向（优先级排序）

1.  **范围预判**：开发轻量分类器（或规则 + embedding 聚类）提前判断查询所属数据库，目标覆盖 70%+ 查询
2.  **重排规模控制**：Determined 场景 Top-100~120 重排；Not Determined 场景禁用或 Top-50
3.  **批量推理**：增大 reranker batch_size（RTX 4060 可支持 32–64）
4.  **结果缓存**：相同 query / db_scope 命中缓存，降低重复计算

### 4.3 长期优化方向

| 优化点 | 预期收益 | 实现难度 | 优先级 |
| --- | --- | --- | --- |
| RRF / Weighted RRF 作为默认融合 | +2–5% MRR，延迟更稳定 | 低   | ★★★★★ |
| Schema 摘要 / 压缩 | 显著降低 LLM token 消耗 | 低   | ★★★★ |
| 端到端指标（Exec Acc） | 真正贴近业务（SQL 执行正确率） | 中   | ★★★★ |
| 知识图谱 / 外键增强 | 提升多表 & 复杂查询能力 | 中   | ★★★ |
| LLM Fine-tuning (Spider/BIRD) | +10–25% Exec Acc | 高   | ★★★ |
| 并行召回 + 异步重排 | 整体延迟 -30–50% | 中   | ★★★ |
