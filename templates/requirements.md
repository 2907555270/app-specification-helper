# 详细需求文档

## 1. 业务需求

### 1.1 背景故事
在大型企业环境中，数据库通常包含数千张表。当用户（开发人员、数据分析师、业务人员）想要查询数据时，往往不知道或不确定应该使用哪些表。现有的nl2sql方案在表数量较少时表现良好，但在面对3000张表这样的海量元数据时，直接将所有表信息塞入prompt中会导致：
- 输入上下文过长，LLM处理效率低下
- 相关表信息被淹没在大量噪声中
- 召回准确率显著下降

因此，需要一个专门的表召回模块，从海量表中精准定位用户意图相关的表。

### 1.2 用户画像
**主要用户群体：**
- **后端开发者**：需要根据业务需求查询数据，但不清楚具体表结构
- **数据分析师**：需要从数据库中提取数据进行分析
- **业务人员**：有数据分析需求但不懂SQL
- **其他系统**：作为nl2sql pipeline的核心组件被调用

**用户痛点：**
1. 不知道数据在哪张表中
2. 表名和字段名难以理解其含义
3. 数千张表无法逐一浏览

### 1.3 核心业务流程
```
用户输入自然语言问题 → 翻译（可选：中文→英文） → 混合模式表召回 → 返回相关表列表
```

**详细流程：**
1. 用户输入自然语言问题（如"哪些球员的得分超过1000？"）
2. 系统将问题转换为英文（可选，取决于原问题语言）
3. **混合召回**：
   - 关键词匹配：从表名、字段名中匹配关键词
   - 稀疏向量召回：利用ES的文本相似度检索
   - 稠密向量召回：利用embedding的语义相似度检索
4. 对三种召回结果进行加权融合
5. 返回排序后的相关表列表

### 1.4 关键业务指标 (KPI)
| 指标 | 目标值 | 优先级 |
|------|--------|--------|
| 表召回准确率 (Hit Rate @K=3) | >= 85% | P0 |
| 表召回准确率 (Hit Rate @K=5) | >= 90% | P1 |
| MRR (Mean Reciprocal Rank) | >= 0.80 | P1 |

### 2.2 详细功能说明

#### 功能：混合模式表召回
**功能描述：**
采用关键词匹配、稀疏向量、稠密向量三种方式综合召回相关表，通过加权融合得到最终排序结果。

**技术方案：**
1. **关键词召回**：
   - 提取问题中的关键词（名词、实体）
   - 在表名、字段名中进行精确/模糊匹配
   - 计算匹配得分

2. **稀疏向量召回**：
   - 将问题向量化（调用稀疏向量API）
   - 在ES中检索相似文档
   - ES索引包含：db_name, table_name, column_name的组合文本

3. **稠密向量召回**：
   - 将问题向量化（调用稠密向量API）
   - 在ES中检索相似文档（需配置dense_vector字段）
   - 使用余弦相似度计算

4. **结果融合**：
   - 对三种召回结果取top-N
   - 加权求和：Score = α × 关键词得分 + β × 稀疏得分 + γ × 稠密得分
   - 按得分排序返回

**边界情况：**
- 问题为空：返回错误提示
- 无匹配结果：返回空列表或最相关的几个表
- 服务不可用：返回降级结果或错误

**数据需求：**
- 输入：自然语言问题（字符串）
- 输出：相关表列表（List[Dict]，包含db_name, table_name, score）
- 存储：ES索引（需要预先构建）

## 4. 数据模型

### 4.1 元数据结构

#### 表元数据（来自metadatas.json）
每条记录格式：`db_name\ntable_name\ncolumn_name1\ncolumn_name2\n...`

```
soccer_3
club
club_id
club_name
...
```

#### 召回结果结构
```python
class TableMatch(BaseModel):
    db_name: str
    table_name: str
    score: float
    match_type: str  # "keyword", "sparse", "dense", "hybrid"
```

#### 批量测试结果
```python
class EvaluationResult(BaseModel):
    total_queries: int
    hit_rate_at_1: float
    hit_rate_at_3: float
    hit_rate_at_5: float
    mrr: float
    details: List[Dict]  # 每个query的召回结果
```

### 4.2 实体关系
- 表元数据是静态数据，启动时加载
- 召回结果动态生成

### 6.1 潜在风险
